AI/ML Portfolio Requirement

Artifact 1: Evaluating an AI-Generated Phishing Analysis

A fictional and sanitized cybersecurity exercise demonstrating how AI can assist a SOC analyst without replacing human verification.

Purpose

Connect AI learning with real-world SOC decision-making

This artifact evaluates whether an AI-generated security analysis correctly identifies a phishing attempt, explains the evidence, recommends an appropriate severity, and proposes safe response actions. No real organizational data, employee information, internal logs, or confidential incidents are used.

Step 1

Fictional Incident Scenario

A user receives an urgent email claiming that their Microsoft 365 password will expire within 30 minutes. The message asks the user to sign in through a shortened URL. The display name is “Microsoft Security,” but the sender domain is unrelated to Microsoft.

Step 2

Prompt Submitted to the AI Tool

Act as a SOC analyst. Analyze this fictional security incident.

Identify:
1. The likely incident classification
2. Indicators of compromise
3. Recommended severity
4. Immediate containment actions
5. Investigation steps
6. User guidance
7. Limitations or assumptions

Step 3

AI Analysis Summary

  • Classified the message as a likely credential-phishing attempt.
  • Identified urgency, sender-domain mismatch, and shortened URL.
  • Recommended isolating the message and blocking the URL.
  • Suggested checking sign-in activity and contacting the user.
  • Noted that severity depends on whether the link was clicked.

Step 4

Human Analyst Evaluation

What the AI did well

It recognized common phishing indicators and proposed reasonable first-response actions.

What required improvement

A final severity should not be assigned until the analyst knows whether the user clicked the link, entered credentials, or generated suspicious authentication activity.

Required analyst validation

Review message headers, URL reputation, email gateway data, endpoint activity, sign-in logs, and whether other users received the same message.

Reflection

What I Learned

This activity reinforced that AI can accelerate the first stage of security analysis by organizing indicators, suggesting investigation steps, and presenting a structured response. However, an AI response should not be treated as a final incident decision. Effective SOC work still requires evidence validation, organizational context, analyst judgment, privacy awareness, and documented response procedures.

The exercise also strengthened my ability to create precise prompts, evaluate AI output critically, and communicate security reasoning to both technical and nontechnical audiences.

AI/ML Portfolio Requirement

Artifact 2: AI and Machine Learning in Cybersecurity Timeline

A research and communication artifact showing how artificial intelligence evolved from rule-based security systems to modern machine-learning and generative-AI applications.

Purpose

Demonstrate research, synthesis, and audience-focused communication

My first artifact focused on evaluating an AI-generated phishing analysis. This second artifact demonstrates a different skill: researching major developments, selecting relevant milestones, and presenting technical history clearly for cybersecurity students, educators, and hiring managers.

Project Context

Why I Selected This Artifact

Artificial intelligence now supports alert prioritization, behavioral analytics, malware detection, phishing analysis, and security automation. Understanding how these capabilities developed helps security professionals evaluate modern tools and their limitations.

Skills Demonstrated

Research Technical Writing Information Synthesis Visual Communication Cybersecurity Awareness Responsible AI

Target Audience

Who This Artifact Is For

  • AI/ML educators reviewing applied learning outcomes.
  • Cybersecurity hiring managers assessing communication skills.
  • SOC analysts learning how AI capabilities evolved.
  • Students seeking a concise introduction to AI in security.
1980s–1990s

Rule-Based Expert Systems

Early security tools relied on predefined rules and signatures. These systems recognized known patterns but struggled with new or changing threats.

Foundation: rule-based reasoning
2000s

Statistical Detection and Log Analysis

Security teams centralized logs and used statistical methods to identify unusual events across networks, endpoints, and applications.

Growth: SIEM and anomaly detection
2010s

Machine Learning for Behavioral Analytics

Machine-learning models began supporting malware classification, user behavior analytics, fraud detection, and alert prioritization.

Expansion: predictive analytics
2020–2022

Automation and AI-Assisted SOC Workflows

Security platforms combined analytics with automation to enrich alerts, correlate evidence, recommend actions, and reduce repetitive analyst work.

Operations: orchestration and triage
2023–Present

Generative AI and Security Copilots

Generative AI can summarize incidents, explain queries, draft investigation steps, and help analysts navigate security data. Human validation remains necessary because results may be incomplete or inaccurate.

Current focus: human-AI collaboration

Challenge

Choosing Meaningful Milestones

The main challenge was avoiding a timeline that simply listed technologies. I selected milestones that show a clear progression: fixed rules, statistical analysis, machine learning, automation, and generative AI.

Solution

Connecting History to SOC Work

Each milestone explains what changed and how it affected security monitoring, alert investigation, and analyst decision-making.

Ethical Consideration

Responsible Use of AI

AI-assisted security tools should be evaluated for accuracy, privacy, bias, transparency, and operational risk. Automated recommendations must be supported by evidence and procedures.

Reflection

What I Learned from Artifact 2

Developing this timeline improved my ability to research a broad technical topic and convert it into a concise story for a specific audience. I learned that a strong portfolio artifact should provide context, explain why the work matters, and show the decisions made during development.

This artifact complements my first artifact because it demonstrates research and communication rather than incident evaluation. Together, the two artifacts show both practical cybersecurity reasoning and the ability to explain AI/ML developments clearly.
